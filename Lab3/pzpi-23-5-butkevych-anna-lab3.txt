2 ЗАВДАННЯ


На цій лабораторній роботі потрібно розробити front-end частину програмної системи. Ця частина системи повинна підтримувати роботу клієнтів у веб-режимі та забезпечувати адміністрування системи, тобто мати окремий інтерфейс для звичайних користувачів і окремий — для адміністраторів (відповідно до ролей користувачів вашої системи). 
Користувацький інтерфейс та інтерфейс адміністрування повинні підтримувати інтернаціоналізацію та локалізацію, а саме, забезпечувати: підтримку української на англійської мов сайту, формат дати та часу відповідний регіону, порядок сортування текстових значень, різні напрями введення тексту тощо. 
Адміністрування системи повинно забезпечувати: управління користувачами системи, управління даними системи, створення резервних копій налаштувань та даних, експорт та імпорт даних та налаштувань.








3 ОПИС ВИКОНАНОЇ РОБОТИ


Назва інструменту та версія: GPT-5 mini – мовна модель штучного інтелекту, використана для синтезу технічного опису, швидкої ітерації варіантів діаграм та формулювання академічного тексту звіту.
Мета використання: допомога у формулюванні структурованого технічного опису, генерації варіантів тексту для звіту та уточненні архітектурних рішень на основі огляду репозиторію.
Внесок автора: Провів огляд кодової бази фронтенду, ідентифікував ключові компоненти, viewmodel-и, контексти та точки взаємодії з бекендом. Сформував архітектурну модель та прийняв інженерні рішення щодо центральних механізмів (AuthContext, централізований `api.ts`, MVVM-подібна організація з ViewModel-ами).


Інженерні рішення й технічні міркування
- Архітектурний стиль: фронтенд побудовано в стилі розділення обов'язків — UI (Pages/Components), стан і бізнес-логіка (ViewModel-и), інфраструктурний шар (`api.ts`, `AuthContext`). Така організація відповідає MVVM-подібному підходу, що сприяє модульності й тестованості.
- State management: застосунок використовує локальний стан у ViewModel-ах і централізований контекст автентифікації (`AuthContext`) для збереження токену й інформації про поточного користувача. Це дозволяє мінімізувати розповсюдження стейту між компонентами і забезпечити однорідний спосіб авторизації запитів.
- API-шар: всі HTTP-виклики інкапсульовані в `api.ts`, що дозволяє централізувати заголовки, обробку помилок мережі, та підміняти реалізацію під час тестування (mocks, fakes).
- UI-компоненти: `components/` містить реюзабельні блоки (Layout, віджети). Сторінки підключають ці компоненти композиційно, що забезпечує узгоджене відображення та спрощує підтримку стилів.
- Міжнародна локалізація: використовується модуль `i18n.tsx` (інтерфейс `useI18n()`), що дозволяє легко локалізувати текст контенту та робити інтерфейс багатомовним.
- Стилі й доступність: стилі застосовано через CSS (файли `.css`); варто перевірити контраст, фокус-керування і семантичну розмітку для покращення доступності.
- Логування та помилки: ViewModel-и мають просту обробку помилок (вивід повідомлень у UI); рекомендовано централізувати логування помилок і додати telemetry для помилок рантайму.
- Тестування: архітектура сприяє юніт-тестуванню ViewModel-ів і інтеграційному тестуванню сторінок із моками для `api.ts`.
- Збірка й деплой: використовуються стандартні інструменти (Vite/npm). Для production- деплою рекомендується статична генерація артефактів та CDN для статики.


Взаємодія компонентів
- Ініціалізація: `main.tsx` → `App.tsx` → `AuthContext` (спроба відновлення сесії) → маршрути → відповідні Pages.
- Приклад робочого циклу:
  1. Користувач відкриває додаток; `AuthContext` відновлює токен (якщо є) та надає його ViewModel-ам.
  2. `DashboardPage` ініціює `useDashboardViewModel()` → ViewModel робить запити через `api.ts` для отримання переліку пристроїв, читань та алертів.
  3. Дані, що повертаються, трансформуються ViewModel-ом і передаються у Page/Components для рендеру.
  4. Дії користувача (напр., фільтрація, підтвердження алерту) викликають методи ViewModel-а, які оновлюють стан і/або відправляють запити на бекенд.


Безпека та права доступу
- Авторизація: токен передається у заголовку авторизації запитів; `AuthContext` управляє зберіганням/оновленням. Потрібно перевірити безпечне зберігання токенів (httpOnly cookie vs localStorage) залежно від вимог проекту.
- Розмежування прав: рольова модель (`Admin`, `Super Admin`) повинна бути підтверджена на бекенді; фронтенд лише приховує або показує інтерфейсні елементи залежно від ролі.

        Опис створених діаграм (Додаток Б):


1. Use Case Diagram
Діаграма прецедентів відображує взаємодію трьох категорій користувачів з системою: звичайний `End User`, `Admin` та `Super Admin`. Система розділена на прямокутник, що позначає межу фронтенд‑додатку.
Для `End User` визначено основні сценарії: Login/Authenticate (вхід у систему з автентифікацією), View Dashboard (перегляд головної інформаційної сторінки), Browse Devices (перегляд списку пристроїв), View Readings (перегляд historії вимірювань), Subscribe to Device (підписка на певний пристрій), View Alerts (перегляд сигналізацій), View Health (перегляд здоров'я/стану пристроїв), Manage Subscriptions (управління підписками).
Для `Admin` прецеденти включають Login, View Dashboard, Admin: Manage Devices (управління пристроями у системі), View Alerts. Admin має розширені права для адміністрування пристроїв.
Для `Super Admin` — найвищий рівень доступу: Login, View Dashboard, Admin: Manage Devices, Manage Users (управління користувачами системи), System Settings (налаштування системи), View Alerts. Super Admin може змінювати конфігурацію та керувати іншими користувачами.
Відношення <<include>> показують залежності: View Readings включає Browse Devices (для перегляду читань потрібно обрати пристрій), View Health також включає Browse Devices, Manage Subscriptions включає Subscribe to Device. Такі зв'язки моделюють логічні послідовності дій у фронтенді.


2. Component Diagram 
Діаграма компонентів показує фізичну структуру фронтенду та його взаємозв'язки. На вершині — основні файли: `App.tsx` (кореневий компонент) та `main.tsx` (точка входу).
Пакет Pages містить вісім сторінок: LoginPage (сторінка входу), DashboardPage (дашборд), DevicesPage (список пристроїв), ReadingsPage (читання), AlertsPage (алерти), HealthPage (здоров'я), SubscriptionsPage (підписки), AdminPage (адмін-панель) та SuperAdminPage (суперадмін-панель для управління користувачами й налаштуваннями).
Пакет ViewModels містить восьмь основних моделей представлення: useAuthViewModel, useDashboardViewModel, useDevicesViewModel, useReadingsViewModel, useAlertsViewModel, useHealthViewModel, useSubscriptionsViewModel та useSuperAdminViewModel. Кожна ViewModel хука управляє станом і логікою для відповідної сторінки.
Пакет UI Components — Layout (шаблон розташування) та Common Widgets (загальні віджети для переконавців елементів).
Контекст — AuthContext, що забезпечує автентифікацію й авторизацію.
Центральне місце займає api.ts — модуль для всіх HTTP-запитів. Він отримує виклики від ViewModels та комунікує з Backend API через REST. AuthContext надає функції і дані для ViewModel-ів.
Визначено залежності: Pages → ViewModels (сторінки використовують ViewModel-и), Pages → Components (сторінки композиційно побудовані з компонентів), ViewModels → api.ts (ViewModel-и витягають дані через API), AuthContext → ViewModel-и (контекст надає дані про авторизацію).


3. Package Diagram ([diagrams/package.puml](diagrams/package.puml))
Діаграма пакетів демонструє логічне розбиття кодової бази у каталозі `src` на п'ять основних пакетів.
pages: містить вісім файлів сторінок (LoginPage, DashboardPage, DevicesPage, ReadingsPage, AlertsPage, HealthPage, SubscriptionsPage, AdminPage, SuperAdminPage). Кожен файл експортує React-компонент функції, що рендерить інтерфейс для відповідного розділу.
viewmodels: містить восьмь файлів хук-ViewModel-ів (useAuthViewModel, useDashboardViewModel, тощо). Кожен хук управляє станом та логікою для однієї або кількох сторінок.
components: вміщує реюзабельні UI-елементи — Layout (розташування) та shared components (загальні віджети), які використовуються кількома сторінками.
context: містить AuthContext.tsx — контекст для управління сесією користувача та авторизацією.
assets: пустой каталог для статичних ресурсів (зображення, іконки, шрифти).
Коренева папка `src` також містить api.ts (HTTP-шар), i18n.tsx (багатомовність), основні CSS-файли та типи.
Залежності: pages → viewmodels (сторінки importують хуки), viewmodels → api.ts (ViewModel-и роблять запити), pages → components (сторінки використовують компоненти), pages → context (сторінки мають доступ до контексту через React Context API), components → assets (компоненти можуть завантажувати статику).


4. Activity Diagram ([diagrams/activity.puml](diagrams/activity.puml))
Діаграма діяльності описує типовий сценарій запуску додатку й входу користувача в систему.
Сценарій починається з відкриття додатку (main.tsx / App.tsx завантажується браузером). Перший крок — перевірка наявності локального токена автентифікації (зберігається у localStorage або sessionStorage). Якщо токен існує, `AuthContext` відновлює сесію автоматично, мінуючи LoginPage, і переходить до завантаження дашборду. Якщо токена немає, користувач отримує LoginPage, вводить облікові дані (login/password), які відправляються через api.ts на бекенд endpoint `/auth/login`.
Після успішної автентифікації (статус 200), токен зберігається в AuthContext та локальному сховищі, сесія відновлюється. У разі помилки (4xx/5xx) виводиться повідомлення про помилку й процес зупиняється.
Після успішного входу система ініціалізує DashboardViewModel, робить запити до api.ts для отримання списку пристроїв, читань та алертів, та рендерить DashboardPage з отриманими даними.
Діаграма включає гілку для `Super Admin`: якщо користувач має роль Super Admin, система додатково увімкнює дії управління користувачами й системними налаштуваннями, надаючи доступ до адміністративних функцій.






4 ВИСНОВКИ
        Розроблений фронтенд-додаток для моніторингу радіаційних пристроїв є архітектурно обґрунтованою системою, що поєднує сучасні підходи веб-розробки з чіткою організацією коду. Система розроблена з використанням React + TypeScript, забезпечуючи типобезпеку й покращену розробник-досвід, чіткий поділ на UI-шар (Pages/Components), логічний шар (ViewModels) та інфраструктурний шар (api.ts/AuthContext) робить код легким для розуміння, тестування та розширення, поєднання локального стану у ViewModel-ах з контекстом автентифікації дозволяє ефективно управляти даними без занадто складних глобальних store-ів, які можуть стати вузьким місцем при зростанні проекту, вбудована архітектурно підтримка трьох ролей (End User, Admin, Super Admin) забезпечує гнучкість у розмежуванні прав доступу й можливосте, використання `i18n` модуля дозволяє легко адаптувати інтерфейс до різних мов, що важливо для глобальних проектів.

useDevicesViewModel:
1 import { useState, useEffect, useCallback } from "react";
2 import {
3   getDevices,
4   getDevice,
5   createDevice,
6   updateDevice,
7   deleteDevice,
8 } from "../api";
9 import type { Device } from "../types";
10 
11 export function useDevicesViewModel() {
12   const [devices, setDevices] = useState<Device[]>([]);
13   const [isLoading, setIsLoading] = useState(true);
14   const [error, setError] = useState("");
15 
16   // Form state
17   const [showForm, setShowForm] = useState(false);
18   const [editingDevice, setEditingDevice] = useState<Device | null>(null);
19   const [formName, setFormName] = useState("");
20   const [formSaving, setFormSaving] = useState(false);
21 
22   // Copy token feedback
23   const [copiedToken, setCopiedToken] = useState<string | null>(null);
24 
25   const load = useCallback(async () => {
26     setIsLoading(true);
27     setError("");
28     try {
29       const res = await getDevices();
30       setDevices(res.devices ?? []);
31     } catch (err) {
32       setError(err instanceof Error ? err.message : "Failed to load devices");
33     } finally {
34       setIsLoading(false);
35     }
36   }, []);
37 
38   useEffect(() => {
39     load();
40   }, [load]);
41 
42   const openCreate = () => {
43     setEditingDevice(null);
44     setFormName("");
45     setShowForm(true);
46   };
47 
48   const openEdit = (device: Device) => {
49     setEditingDevice(device);
50     setFormName(device.name);
51     setShowForm(true);
52   };
53 
54   const closeForm = () => {
55     setShowForm(false);
56     setEditingDevice(null);
57     setFormName("");
58     setError("");
59   };
60 
61   const save = async (e: React.FormEvent) => {
62     e.preventDefault();
63     if (!formName.trim()) {
64       setError("Введіть назву пристрою");
65       return;
66     }
67     setFormSaving(true);
68     setError("");
69     try {
70       if (editingDevice) {
71         await updateDevice(editingDevice.id, { name: formName });
72       } else {
73         await createDevice(formName);
74       }
75       await load();
76       closeForm();
77     } catch (err) {
78       setError(err instanceof Error ? err.message : "Failed to save");
79     } finally {
80       setFormSaving(false);
81     }
82   };
83 
84   const toggleActive = async (device: Device) => {
85     setError("");
86     try {
87       await updateDevice(device.id, { is_active: !device.is_active });
88       await load();
89     } catch (err) {
90       setError(err instanceof Error ? err.message : "Failed to update");
91     }
92   };
93 
94   const remove = async (id: string) => {
95     setError("");
96     try {
97       await deleteDevice(id);
98       await load();
99     } catch (err) {
100       setError(err instanceof Error ? err.message : "Failed to delete");
101     }
102   };
103 
104   const copyToken = (token: string) => {
105     navigator.clipboard.writeText(token);
106     setCopiedToken(token);
107     setTimeout(() => setCopiedToken(null), 2000);
108   };
109 
110   return {
111     devices,
112     isLoading,
113     error,
114     showForm,
115     editingDevice,
116     formName,
117     setFormName,
118     formSaving,
119     openCreate,
120     openEdit,
121     closeForm,
122     save,
123     toggleActive,
124     remove,
125     copiedToken,
126     copyToken,
127     refresh: load,
128   };
129 }
