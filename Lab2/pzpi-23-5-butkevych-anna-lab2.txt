2 ЗАВДАННЯ


На цій лабораторній роботі потрібно розробити мобільний програмний застосунок для будь-якої поширеної на сьогодні мобільної платформи. Мобільний застосунок повинен бути частиною системи і повинен взаємодіяти з нею. Для реалізації програмного застосунку можна використовувати технології для реалізації програмного забезпечення для мобільних платформ, вказані лектором, а саме Kotlin для Android та Swift для iOS.
Звіт повинен містити опис прийнятих інженерних рішень. У ньому повинна бути показана будова мобільного застосунку та його взаємодія  іншими частинами системи.








3 ОПИС ВИКОНАНОЇ РОБОТИ


Для підготовки технічної документації та структурування матеріалу було використано сервіси штучного інтелекту.


Назва інструменту та версія: GPT-5.4 mini
Мета використання: допомога у формуванні технічного опису, структуруванні звіту, узгодженні архітектурних рішень та підготовці UML-моделей у форматі PlantUML.
Внесок автора: автор самостійно визначив архітектурну концепцію мобільного застосунку, сформулював вимоги до взаємодії з сервером, структурував пакети проєкту, перевірив відповідність UML-діаграм фактичній структурі коду та адаптував текст звіту до вимог лабораторної роботи.


Система являє собою мобільний клієнт для платформи моніторингу радіаційних показників. У ході виконання лабораторної роботи було розроблено архітектуру Android-застосунку мовою Kotlin із використанням Jetpack Compose для побудови інтерфейсу користувача.
Мобільний застосунок є частиною загальної клієнт-серверної системи та забезпечує взаємодію користувача з такими функціями, як авторизація, перегляд пристроїв, сповіщень, показань, підписок і налаштувань. Усі дані отримуються з серверної частини через REST API, а частина службової інформації зберігається локально.
Опис прийнятих інженерних рішень:
* архітектурний патерн MVVM:
* Застосунок побудовано за патерном Model–View–ViewModel.
* Це дозволяє розділити:
   * View — екрани Jetpack Compose;
   * ViewModel — логіку обробки дій користувача та стану UI;
   * Model / Repository — доступ до даних і взаємодію з сервером.
* декларативний UI на Jetpack Compose:
Усі екрани застосунку реалізовані як composable-компоненти. Це спрощує оновлення інтерфейсу на основі стану та робить UI більш гнучким і модульним.
* централізована навігація:
Для переходів між екранами використано Navigation Compose.
* Навігація організована через маршрути Screen.Auth та Screen.Main, що дозволяє окремо керувати потоками авторизації та основної частини застосунку.
* мережева взаємодія з бекендом:
Для роботи з сервером застосовано Retrofit і REST API.
Через ApiService мобільний клієнт отримує:
   * дані про пристрої;
   * показання датчиків;
   * сповіщення;
   * підписки;
   * інформацію про користувача.
* автентифікація через JWT-токен:
Після успішного входу токен зберігається локально, а AuthInterceptor автоматично додає його до більшості HTTP-запитів. Це забезпечує прозору авторизацію без ручного передавання токена в кожному запиті.
* локальне збереження налаштувань:
Для збереження мови інтерфейсу використано Preferences DataStore. Це дозволяє зберігати вибір користувача між запусками застосунку.
* локальне сховище на Room:
У проєкті передбачено Room Database з AppDb, DAO та entity-класами.
Це створює основу для локального зберігання структурованих даних та розширення функціональності у майбутньому.
* Hilt для ін’єкції залежностей:
Для керування залежностями використано Hilt.
Через модулі AppModule, NetworkModule і RepositoryModule конфігуруються:
   * DataStore;
   * Retrofit;
   * ApiService;
   * репозиторії;
   * база даних Room.
* фасадний репозиторій MainRepository:
MainRepository об’єднує окремі репозиторії:
   * AuthRepository
   * DeviceRepository
   * ReadingRepository
   * AlertRepository
   * SubscriptionRepository
Такий підхід спрощує використання джерел даних у ViewModel і робить архітектуру більш зрозумілою.
Мобільний застосунок виступає клієнтською частиною системи.
Логіка взаємодії побудована таким чином: користувач працює з UI, UI генерує Actions, які передаються у ViewModel, після чого ViewModel виконує звернення до Repository. Репозиторії формують запити до серверної частини, а отримані відповіді оновлюють стан екранів у застосунку.
Таким чином, мобільний застосунок не лише відображає дані, але й забезпечує повноцінну взаємодію з серверною інфраструктурою системи моніторингу.
Для наочного представлення структури, поведінки та архітектури розробленого мобільного клієнта було побудовано такі графічні моделі (Додаток Б):
* UML діаграма прецедентів (Use Case Diagram):
Відображає основні сценарії використання застосунку: авторизація, реєстрація, перегляд дашборду, пристроїв, показань, сповіщень, підписок, зміна мови та вихід із системи.
* UML діаграма компонентів (Component Diagram):
Показує внутрішню будову застосунку та взаємозв’язки між UI, ViewModel, репозиторіями, мережею, локальним сховищем і базою даних.
* UML діаграма діяльності (Activity Diagram):
Описує послідовність дій під час запуску застосунку та проходження користувачем сценарію авторизації.
* UML діаграма пакетів (Package Diagram):
Демонструє логічну структуру проєкту та розподіл коду на пакети ui, repository, network, db, di, navigation, config, data, extension, location.


4 ВИСНОВКИ
У результаті виконання лабораторної роботи було успішно спроєктовано та задокументовано архітектуру мобільного застосунку для системи моніторингу радіаційних показників. Було обґрунтовано вибір інженерних рішень, зокрема використання Kotlin, Jetpack Compose, MVVM, Retrofit, Hilt, Room та DataStore, а також змодельовано взаємодію між мобільним клієнтом і серверною частиною системи.
Побудовані UML-моделі відображають ключові аспекти роботи застосунку: сценарії використання, внутрішню компонентну структуру, основний алгоритм роботи інтерфейсу та логічну структуру пакетів. Це дозволяє повноцінно представити мобільний застосунок як частину цілісної системи та підтверджує коректність прийнятих архітектурних рішень.
